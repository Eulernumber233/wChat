# FileServer Design Document

> wChat file transfer module design.
> This document serves as the implementation blueprint for the file
> management subsystem. It covers architecture, protocols, data
> structures, detailed message flows, and a phased implementation plan.

---

## 1. Overview

### 1.1 Goal

Add a dedicated **FileServer** to handle file upload / download between
clients, starting with **image messages**. The ChatServer remains the
coordination hub — it registers file metadata, issues one-time tokens,
and notifies recipients. The FileServer is a pure storage worker: it
receives files, writes them to disk, and serves them back on request.

### 1.2 Design Principles

- **Separation of concerns**: chat messages flow through ChatServer,
  file bytes flow through FileServer. They never mix on the same TCP
  connection.
- **On-demand connection**: clients connect to FileServer only when
  uploading or downloading. No persistent connection required.
- **One-time tokens**: every upload / download is authorised by a
  short-lived, single-use token generated by ChatServer and stored in
  Redis. FileServer validates the token, then deletes it immediately.
- **ChatServer as coordinator**: FileServer never directly pushes
  messages to clients. After a file is stored, FileServer notifies
  ChatServer via gRPC; ChatServer writes the chat_messages record and
  pushes the message to the recipient through existing channels.
- **Disk storage**: file bytes go to the local filesystem, not to MySQL.
  MySQL stores only metadata (chat_files table).

## 2. Data Structures

### 2.1 MySQL: New Table `chat_files`

```sql
CREATE TABLE chat_files (
    file_id       VARCHAR(64)  PRIMARY KEY,
    uploader_uid  INT          NOT NULL,
    file_name     VARCHAR(255) NOT NULL,
    file_size     BIGINT       NOT NULL,
    file_type     TINYINT      NOT NULL DEFAULT 0,
    mime_type     VARCHAR(64)  NOT NULL,
    file_path     VARCHAR(512) DEFAULT '',
    md5           VARCHAR(32)  DEFAULT '',
    status        TINYINT      NOT NULL DEFAULT 0,
    upload_time   DATETIME     DEFAULT NOW(),
    INDEX idx_uploader (uploader_uid),
    INDEX idx_status   (status)
);
```

**status values**:

| Value | Name        | Meaning                         |
|-------|-------------|---------------------------------|
| 0     | registered  | ChatServer created the record   |
| 1     | uploading   | FileServer is receiving data    |
| 2     | uploaded    | File is fully stored on disk    |
| 3     | failed      | Upload timed out or was aborted |

**file_type values**:

| Value | Type  |
|-------|-------|
| 0     | image |
| 1     | file  |
| 2     | audio |

### 2.2 MySQL: `chat_messages.msg_type` Extension

| Value | Type  | content format                                     |
|-------|-------|----------------------------------------------------|
| 0     | text  | `[{"msgid":"...","content":"..."}]` (existing)      |
| 1     | image | `{"msgid":"...","file_id":"...","file_name":"...","file_size":N,"file_type":1}` |
| 2     | file  | same structure as image, file_type=2               |
| 3     | audio | same structure as image, file_type=3               |

### 2.3 Redis Keys (new)

| Key                                | Value (JSON)                                        | TTL      | Purpose                        |
|------------------------------------|-----------------------------------------------------|----------|---------------------------------|
| `file_upload_token:<token>`        | `{"file_id":"...","uid":N,"file_size":N}`           | 600s     | One-time upload authorisation   |
| `file_download_token:<token>`      | `{"file_id":"...","uid":N}`                         | 600s     | One-time download authorisation |
| `file_upload_progress:<file_id>`   | `"<bytes_received>"`                                | 3600s    | Resume offset for interrupted uploads |

### 2.4 Client `global.h`: New ReqId Values

```cpp
// ChatServer TCP: file coordination
ID_FILE_UPLOAD_REQ       = 1101,  // C -> ChatServer: request to upload
ID_FILE_UPLOAD_RSP       = 1102,  // ChatServer -> C: file_id + token + addr
ID_FILE_NOTIFY_COMPLETE  = 1103,  // ChatServer -> C(sender): upload done, msg sent
ID_FILE_MSG_NOTIFY       = 1105,  // ChatServer -> C(receiver): file msg + download token

// FileServer TCP: binary transfer
ID_FSVR_AUTH_REQ  = 2001,  // C -> FileServer: authenticate with file_token
ID_FSVR_AUTH_RSP  = 2002,  // FileServer -> C: auth result + resume offset
ID_FSVR_DATA      = 2003,  // C <-> FileServer: file data chunk
ID_FSVR_DONE      = 2004,  // FileServer -> C: transfer complete
```

### 2.5 Protobuf: New `FileService` (in `proto/message.proto`)

```protobuf
message FileUploadDoneReq {
    string file_id   = 1;
    string file_path = 2;
    string md5       = 3;
}
message FileUploadDoneRsp {
    int32 error = 1;
}

message FileDownloadAuthReq {
    string file_id    = 1;
    string file_token = 2;
    int32  uid        = 3;
}
message FileDownloadAuthRsp {
    int32 error = 1;
}

service FileService {
    rpc NotifyUploadDone(FileUploadDoneReq) returns (FileUploadDoneRsp) {}
    rpc RegisterDownloadAuth(FileDownloadAuthReq) returns (FileDownloadAuthRsp) {}
}
```

### 2.6 FileServer TCP Frame Format

```
+----------+-----------+-----------+------------------+
| msg_id   | body_len  | offset    | payload          |
| 2 bytes  | 4 bytes   | 8 bytes   | up to 64 KB      |
| LE       | LE        | LE        | JSON or binary   |
+----------+-----------+-----------+------------------+
```

- `ID_FSVR_AUTH_REQ / RSP`: payload is JSON
- `ID_FSVR_DATA`: payload is raw file bytes; offset marks position in file
- `ID_FSVR_DONE`: payload is JSON `{"file_id":"...","md5":"..."}`

## 3. Message Flows

### 3.1 Upload Flow (Sender)

```
Step  Direction                      Content
----------------------------------------------------------------------

 1    Client                         User picks an image in ChatPage.
      (ChatPage)                     Detects non-text -> file send path.

 2    Client -> ChatServer (TCP)     ID_FILE_UPLOAD_REQ {
                                         fromuid:   1001,
                                         touid:     1002,
                                         file_name: "photo.jpg",
                                         file_size: 1048576,
                                         file_type: 1,
                                         md5:       "abc123..."
                                     }

 3    ChatServer:
      3a. file_id    = generate UUID
      3b. file_token = generate UUID
      3c. INSERT INTO chat_files (
              file_id, uploader_uid, file_name, file_size,
              file_type, mime_type, status=0
          )
      3d. Redis SET file_upload_token:<file_token>
              -> {"file_id":"...","uid":1001,"file_size":1048576}
              EX 600
      3e. Read FileServer host/port from config.ini [FileServer]

      ** ChatServer does NOT write chat_messages yet.
      ** ChatServer does NOT notify the receiver yet.

 4    ChatServer -> Client (TCP)     ID_FILE_UPLOAD_RSP {
                                         error:      0,
                                         file_id:    "a1b2c3d4-xxxx",
                                         file_token: "t1t2t3t4-xxxx",
                                         host:       "127.0.0.1",
                                         port:       "8100"
                                     }

 5    Client:
      5a. Show "uploading..." bubble in ChatView
      5b. Open TCP connection to FileServer

 6    Client -> FileServer (TCP)     ID_FSVR_AUTH_REQ {
                                         file_token: "t1t2t3t4-xxxx",
                                         file_id:    "a1b2c3d4-xxxx"
                                     }

 7    FileServer:
      7a. Redis GET file_upload_token:<file_token>
      7b. Validate file_id matches
      7c. Redis DEL file_upload_token:<file_token>  (one-time consume)
      7d. Redis GET file_upload_progress:<file_id>  (resume check)
          -> exists: offset = stored value
          -> absent: offset = 0
      7e. Create / open target file on disk

 8    FileServer -> Client (TCP)     ID_FSVR_AUTH_RSP {
                                         error:  0,
                                         offset: 0
                                     }

 9    Client -> FileServer (TCP)     ID_FSVR_DATA {offset: 0,      data: [64KB]}
                                     ID_FSVR_DATA {offset: 65536,  data: [64KB]}
                                     ...

      FileServer per chunk:
      9a. Write to disk at offset
      9b. Periodically update Redis file_upload_progress:<file_id>

10    FileServer detects total_received == file_size:
      10a. Compute MD5, verify against client's reported md5
      10b. UPDATE chat_files SET status=2,
               file_path='2026/04/09/a1b2c3d4.jpg'
               WHERE file_id='a1b2c3d4-xxxx'
      10c. Redis DEL file_upload_progress:<file_id>

11    FileServer -> Client (TCP)     ID_FSVR_DONE {
                                         file_id: "a1b2c3d4-xxxx",
                                         status:  "ok"
                                     }

12    Client: disconnect from FileServer.
      UI remains "uploading..." until step 14c confirms.

13    FileServer -> ChatServer       gRPC FileService::NotifyUploadDone {
      (gRPC)                             file_id:   "a1b2c3d4-xxxx",
                                         file_path: "2026/04/09/a1b2c3d4.jpg",
                                         md5:       "abc123..."
                                     }

      On gRPC failure: FileServer retries up to 3 times with
      exponential backoff (1s, 2s, 4s).

14    ChatServer:
      14a. UPDATE chat_files SET status=2, file_path=..., md5=...
      14b. INSERT INTO chat_messages (
               conversation_id, sender_id, receiver_id,
               msg_type = 1,
               content  = '{"msgid":"uuid-xxx",
                            "file_id":"a1b2c3d4",
                            "file_name":"photo.jpg",
                            "file_size":1048576,
                            "file_type":1}'
           )
      14c. Notify sender:
           Session->Send(ID_FILE_NOTIFY_COMPLETE, {file_id, error:0})
           Sender UI: "uploading..." -> show sent image

      14d. Check receiver online:
           Redis GET useripprefix_<1002>
           -> offline: STOP. Receiver loads it from history on login.
           -> online: continue to step 15.

15    ChatServer prepares download for receiver:
      15a. download_token = generate UUID
      15b. Redis SET file_download_token:<download_token>
               -> {"file_id":"a1b2c3d4","uid":1002}
               EX 600
      15c. If receiver is on a different ChatServer instance:
           forward via existing ChatGrpcClient (same as text messages)

16    ChatServer -> Client B (TCP)   ID_FILE_MSG_NOTIFY {
                                         fromuid:    1001,
                                         touid:      1002,
                                         msg_type:   1,
                                         msgid:      "uuid-xxx",
                                         file_id:    "a1b2c3d4-xxxx",
                                         file_name:  "photo.jpg",
                                         file_size:  1048576,
                                         file_type:  1,
                                         file_host:  "127.0.0.1",
                                         file_port:  "8100",
                                         file_token: "d5d6d7d8-xxxx"
                                     }
```

### 3.2 Download Flow (Receiver)

```
Step  Direction                      Content
----------------------------------------------------------------------

17    Client B receives ID_FILE_MSG_NOTIFY:
      17a. Check local cache: cache/files/<file_id>.jpg
           -> hit: display image immediately. DONE.
      17b. Cache miss: show "loading..." placeholder in ChatView
      17c. Open TCP connection to FileServer

18    Client B -> FileServer (TCP)   ID_FSVR_AUTH_REQ {
                                         file_token: "d5d6d7d8-xxxx",
                                         file_id:    "a1b2c3d4-xxxx"
                                     }

19    FileServer:
      19a. Redis GET file_download_token:<file_token>
      19b. Validate file_id matches
      19c. Redis DEL (one-time consume)
      19d. Query chat_files for file_path
      19e. Open disk file, prepare to send

20    FileServer -> Client B (TCP)   ID_FSVR_AUTH_RSP {
                                         error:     0,
                                         file_size: 1048576,
                                         file_name: "photo.jpg"
                                     }

21    FileServer -> Client B (TCP)   ID_FSVR_DATA {offset: 0,     data: [64KB]}
                                     ID_FSVR_DATA {offset: 65536, data: [64KB]}
                                     ...
                                     ID_FSVR_DONE {file_id, md5}

22    Client B:
      22a. Write to cache/files/a1b2c3d4.jpg
      22b. Verify MD5
      22c. UI: "loading..." -> display image
      22d. Disconnect from FileServer
```

### 3.3 Offline Receiver (History Load on Login)

No special logic is needed. The existing LoginHandler is extended:

```
Step  Direction                      Content
----------------------------------------------------------------------

A     Client B logs in -> ChatServer LoginHandler loads history
      (existing GetMessages logic reads chat_messages)

B     ChatServer finds a msg_type=1 (image) message:
      B1. download_token = generate UUID
      B2. Redis SET file_download_token:<download_token>
              -> {"file_id":"...","uid":<receiver_uid>}
              EX 600
      B3. Attach file_host, file_port, file_token to the message JSON

C     ChatServer -> Client B (TCP)   ID_CHAT_LOGIN_RSP {
                                         ...existing fields...
                                         friend_list: [{
                                             chat_msgs: [{
                                                 msg_type:   1,
                                                 file_id:    "a1b2c3d4",
                                                 file_name:  "photo.jpg",
                                                 file_size:  1048576,
                                                 file_token: "<new-token>",
                                                 file_host:  "127.0.0.1",
                                                 file_port:  "8100",
                                                 ...
                                             }]
                                         }]
                                     }

D     Client B: for each file message -> check cache -> miss -> download
      (same as steps 17-22)
```

## 4. Token Design Summary

| Token type         | Created by  | Stored in Redis                   | Validated by | Lifetime       | Purpose                        |
|--------------------|-------------|-----------------------------------|-------------- |----------------|--------------------------------|
| Login token        | StatusServer| `usertokenprefix_<uid>`           | ChatServer   | Login session  | User identity (existing)       |
| Upload file token  | ChatServer  | `file_upload_token:<token>`       | FileServer   | 600s, one-time | Authorise upload of one file   |
| Download file token| ChatServer  | `file_download_token:<token>`     | FileServer   | 600s, one-time | Authorise download of one file |

**Key properties**:
- Tokens are UUIDs, bound to a specific file_id + uid
- FileServer DELETEs the token from Redis immediately after validation
- Even if a token leaks, it can only be used once and only for one file
- FileServer has no concept of "logged-in users"; it only knows tokens

## 10. File Naming on Disk

```
<StoragePath>/
  2026/
    04/
      09/
        a1b2c3d4-xxxx.jpg      file_id + original extension
        e5f6g7h8-xxxx.png
      10/
        ...
```

- Directories created by date (year/month/day) to avoid single-directory
  file count limits.
- Files named by file_id (UUID) to prevent collisions. Original extension
  preserved for MIME type inference.
- `chat_files.file_path` stores the relative path:
  `"2026/04/09/a1b2c3d4-xxxx.jpg"`.
  FileServer prepends `StoragePath` from config.ini when accessing disk.
